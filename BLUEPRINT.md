# mc-working-os —— 个人工作操作系统蓝图

## Context（为什么做这个）

用户是数据工程师，日常工作是：把需求转成代码、研究如何完成需求、沉淀项目经验、给同事分享/解释技术。他想搭一个系统，达成两个互相喂养的目标：

1. **把工作做快**：会议/笔记/交流"摄取"成项目 context → 新需求来时 AI 基于积累的 context 快速产出代码与分享材料。系统越用越懂项目、越懂他。
2. **把人沉淀下来**：从每天的对话/代码/分享里复盘 → 反复出现的认知提炼成可带走的"认知资产" → 系统反过来照见他的长处与盲点。

底层目的：把"执行"的时间省出来，留给思考和学习；并攒下离开公司也能带走的认知资产。

**关键决定（已和用户确认）：**
- **agent 中立 / 可移植**：系统内核是一个**自描述的纯 markdown 仓库**，不绑定任何单一 agent。用户会用 **Claude Code、Cursor、Codex、GitHub Copilot** 等多个 agent。行为与知识全在仓库的纯 markdown 里；各 agent 只是可替换的"触发前端"，不持有独有逻辑。换 agent = 换前端，本体一行不改。
- 用户当前想要**完整蓝图**，不急着选先做哪块。下方给出建议建造顺序，可任意切入。

系统根目录：`/home/meiqiao/projects/mc-working-os/`（目前只有 idea.md）。

## 整体架构

分**通用内核**（纯 markdown，任何 agent 都能读）和**各 agent 薄适配器**（只负责触发）。

| 层 | 载体（通用内核） | 作用 |
|---|---|---|
| **入口/约定** | 仓库根 `AGENTS.md`（跨 agent 事实标准） | 讲清系统是什么、关键词有哪些、文件在哪、每个 skill 先读哪个文件、先加载哪些 profile。**唯一真源** |
| **行为层（skills）** | `skills/{摄取,执行,润色,分享,复盘,学习}.md` | 每个 skill = 一份给 AI 的工作流：触发词、流程、读写哪些文件，纯 markdown。任何 agent 读它即可照做 |
| **知识层** | `projects/ learning/ reviews/ assets/ style/` | 项目 context、学习笔记、复盘、认知资产、风格画像——按需加载 |
| **常驻指针** | `AGENTS.md` 内引用 + `assets/INDEX.md` + `style/*-profile.md` | 取代 Claude 独有的自动 memory：由 skill 指令**显式"先读这几个文件"**，在任何 agent 里都成立 |

> 术语说明：**skill = 给 AI 的工作流（`skills/<名>.md`）**，agent 中立的纯 markdown，系统真正的行为都在这里。注意别和 Claude Code 自家的 `.claude/skills/` 目录混——后者只是它的"触发适配器"，内容就一行：重定向到根目录的 `skills/<名>.md`。

**各 agent 薄适配器（一行重定向到 `AGENTS.md`/skill，几乎零重复）：**

| Agent | 适配器 | 说明 |
|---|---|---|
| Codex | 无需 | 原生读 `AGENTS.md` |
| Cursor | `AGENTS.md`（也可加 `.cursor/rules/`） | 原生读 `AGENTS.md` |
| Claude Code | `.claude/skills/<名>/SKILL.md` + `CLAUDE.md`→`AGENTS.md` | 适配器仅一句"读根目录 `skills/<名>.md` 照做" |
| GitHub Copilot | `.github/copilot-instructions.md`→`AGENTS.md` | 一行重定向 |

设计要点：**重内容放知识层（按需读），精炼结论放常驻指针（每会话显式先读）**。常驻指针只存"提炼后能直接当 prompt 用"的规则与摘要，避免撑爆上下文。

仓库 `git init`，让 context/资产/风格画像的演变可追溯（能看到自己认知怎么长出来的）。

## 六个 skill（给 AI 的工作流）

每个 skill 是一个 `skills/<名>.md`（纯 markdown），定义触发词、流程、读写哪些文件——本质就是"这个关键词来了，AI 该走哪套流程"。用户对**任何 agent** 说关键词（如"摄取…"），agent 读过 `AGENTS.md` 后即去加载对应 skill 执行。下面用 `/<名>` 指代关键词触发。

### 1. `/摄取 [项目名]` —— 抓 context（主线一入口）
- 你粘贴会议记录/同事交流/随手笔记。
- 流程：②抽取结构化信息（决策、待办、术语、涉及的人、开放问题）→ ③追加进 `projects/<项目>/context.md`，原文存 `ingest/<日期>.md`。
- 项目不明确时主动问归到哪个项目。

### 2. `/执行 <项目名> <需求>` —— 把活干快（主线一产出）
- 按 skill 指令**先读** `projects/<项目>/context.md` + `decisions.md` + `pitfalls.md` 作为背景。
- 基于积累的 context 把需求转成代码/方案。干活时发现的新决策/新坑**作为副作用自动回写**（执行回写不打断你；见下方"调和"）。
- 因此 `/执行` 可指向真实工作仓库读代码与表结构；mc-working-os 只承载 context/认知层，不复制代码。

### 3. `/学习 <话题>` —— 一起学
- 进入苏格拉底式探讨：和你一来一回把话题讲透、帮你理解。
- 结束时把对话提炼成学习笔记，存 `learning/<话题>/`。

### 4. `/润色` —— 改英语 + 学你的风格（循环 A）
- `/润色` + 粘英文 → 改语法、给地道建议、给一个**已套用 `english-profile.md` 当前风格**的润色版。
- 你针对它的版本再改 → `/润色 收` 把你的改后版喂回 → 系统**对比 [它的版本] vs [你的版本]** 的语义差异（用词、句长、直接/委婉、语气、结构）→ 把差异**提炼成规则**写进 `english-profile.md`，配对存进 `english-samples/`。
- 越用，产出越像你说话。

### 5. `/分享 <主题>` —— 分享初稿 + 学你的分享风格（循环 A 的孪生）
- 用 `sharing-profile.md` 给分享/演示初稿。
- 同样的反馈闭环：你改 → 喂回 → 提炼你的分享风格规则。

**分享后的回流（三个门，防回声室）：** 系统自己的产出不自动变成自己的输入，只有外部世界的反馈才自动回流。
1. **分享稿本身** → 不回灌 context；只通过改稿 diff 喂 `sharing-profile.md`（风格）。
2. **外部反馈**（同事提问、哪里卡住、会后追问）→ 真新信号，**走 `/摄取`** 进项目 context（也被 `/复盘` 捡为反思/盲点材料）。
3. **你讲透的好解释** → **由你手动晋升**进 `learning/<话题>/` 或 context，不自动。

> 原则：系统产出要回流，得你主动按"晋升"；外部反馈才自动进。同样管住润色线。

### 6. `/复盘` —— 沉淀认知（主线二，循环 B）
- 复盘当天：扫今天的对话、写的代码、做的分享、摄取的笔记 → 写 `reviews/daily/<日期>.md`。每条认知**结构化**打标签（type / statement / evidence / 所属情境-项目 / tags），而非纯散文——这是后续聚类的前提。
- **候选池增量聚类**：把当天每条认知拿去和 `reviews/recurring.md` 里的现有聚类做**语义匹配**（命中→count+1、加 instance、更新 last_seen、若新项目则 contexts+1、打磨 canonical；未命中→开新候选 count=1）。不全量重扫历史。
- **晋升判定**（详见下方"循环 B"）：够格的候选**综合提炼+停下来问你确认**后写进 `assets/`，链回证据。
- **照镜子**：定期读 assets + reviews 全集 → 指出你还没命名的长处/盲点 → 写 `mirror.md`。
- 触发为手动（对任意 agent 说"复盘"）；想自动化的 agent（如 Claude Code 定时任务）可选配，但**不是内核依赖**。

## 项目 context 如何保持更新（做快主线的命脉）

**原则：只存代码说不出来的东西。** context.md 不是代码的镜像（那会立刻过时），只存代码里没有的那层：为什么这么设计、谁拍的板、试过什么放弃了、还没解决的开放问题、术语。这层变化慢、且不与代码冲突，能"过时"的表面积天然就小。

**三种载体，三种更新语义：**

| 载体 | 更新方式 |
|---|---|
| `ingest/<日期>.md` 原始笔记 | 只追加，永不改（不可变事件流，可回溯） |
| `decisions.md` 决策日志 | 只追加，旧决策标 `已被 X 取代 (日期)` |
| `context.md` 当前理解 | **每次 reconcile（调和），不是盲目 append** |

**调和：四种操作（不是机械追加）。** 每次摄取/执行产生新信息，对 context.md 每条做判断——①新增 ②更新（改写 live 条目 + 标变更日期）③矛盾（旧的标 superseded、写入新的，保留"曾经这么以为"）④重复（跳过）。由 AI 做语义合并。

**人机边界（已和用户确认）：关键处停下来问，其余自动。**
- **矛盾**（新旧冲突）和**归档**（把某条移出 live 区）这两类不可逆判断 → **停下来问用户确认**。
- 其余（新增、纯更新、跳过重复）→ 自动。
- 特例：`/执行` 的副作用回写**全自动不打断**（你在干活，别被打断）；`/摄取` 外部信息在矛盾点才问。

**context.md 按更新频率分区：** 一句话(稳定) / 术语表(只增) / 关键人(偶改) / 设计的"为什么"(演进带日期) / 当前状态 live(高频 reconcile) / 开放问题(状态 open→resolved，resolved 归档) / → decisions.md、pitfalls.md。

**三个更新来源（让它自动保鲜）：** `/摄取`（外部信息）、`/执行`（工作副产品自动回写）、`/复盘`（兜底捡漏）。

**压实（compaction）：** 挂在 `/复盘` 周档或单独 `/整理 <项目>`——context.md 超长时，把已 resolved 的开放问题、已 superseded 的旧状态归档回 `ingest/`，保持 live 视图精简。

## 两个核心循环（系统的灵魂，唯一需从头设计的部分）

**循环 A：靠 diff 学风格（润色/分享共用）**
- 关键是 `*-profile.md` 必须是**规则化、可直接当 prompt 注入**的（例："偏好短句""避免被动语态""喜欢用具体例子开头"），而不是堆原始 diff。
- 机制：产出 v0（用当前 profile）→ 你改成 v1 → 喂回 → 提炼差异成规则追加进 profile → 下次产出注入 profile。

**循环 B：判定"反复出现"才算认知资产（已和用户细化）**

核心认知：**"反复出现"不是字符串计数，而是语义聚类**——同一条认知每次措辞都不同（"先验证数据再动手"="又没看清上游就开干"）。判定真正的难点在"这两条算不算同一条"，由 AI 在写入候选池那一刻、带上下文判断。

*纯频率既太严（错过单次重磅顿悟）又太松（平凡的"今天累"也高频）。真正让资产"可带走"的信号不是频率，而是**跨情境复现**。*

**候选池 `reviews/recurring.md`**（增量聚类，不全量重扫）。每条聚类：
```
- canonical: 规范化的一句话认知（打磨到"可复用原则"高度）
  type: 认知 / 坑 / 方法 / 偏好
  count: 3   first_seen / last_seen   contexts: [项目A, 项目C]
  instances: [每次的原话 + 链回 reviews/daily/<日期>.md]
  status: 候选 / 已晋升 / 已搁置
```

**晋升门槛 = 平衡型 + 跨情境硬性要求（用户已定），两扇门：**
- **门一·复现型**：`≥3 个不同日子` **且** `跨 ≥2 个情境/项目`（**硬门**——单项目内的反复只进该项目 context，不进资产库）**且** 过非平凡门槛 → 标签 `已验证可迁移`。
- **门二·单次高信号型**：被你（或系统）显式标记"这条重要"的深刻顿悟 → 绕过复现/跨情境，过非平凡门槛即可 → 标签 `待验证`（可迁移性尚未被证明，诚实标注）。
- **非平凡门槛**：必须是关于"你怎么思考/工作"的，不是一时状态或情绪。

**晋升 = 综合提炼 + 你确认（不是复制原话）：** 把聚类拔高成一句脱离单次实例的资产 → 归类(心智模型/坑/长处盲点) → **停下来问你**(接受/改写/否决/搁置，与"关键处问我"一致) → 写进 `assets/` 并链回证据 → 候选标 `已晋升`。

**衰减：** 长期未复现的候选标 `已搁置`（休眠不删），保留你曾经的关注点；`mirror.md` 照镜子会用到——它可能正是你已成长过去的阶段。

## 建议建造顺序（可任意切入）

- **Phase 0 — 骨架**：建目录结构、`git init`、写 `AGENTS.md`（唯一真源）+ 空 `skills/` + 4 个 agent 的薄适配器（Claude Code `.claude/skills/` + `CLAUDE.md`、Cursor、Copilot；Codex 零配）+ 常驻指针约定。
- **Phase 1 — 做快主线**：`skills/摄取.md` + `执行.md`，立刻有生产力回报。
- **Phase 2 — 风格循环**：`skills/润色.md` + 循环 A（先英语，分享复用同机制）。
- **Phase 3 — 沉淀主线**：`skills/复盘.md` + 循环 B + 认知资产晋升。
- **Phase 4 — 收尾**：`skills/分享.md`、`学习.md`、照镜子、（可选）各 agent 自动复盘。

## 验证方式（每个 skill 端到端测）

每块做完后，用真实但脱敏的样例跑一遍，检查文件是否按预期生成/更新：

1. **摄取**：丢一段会议笔记 → 确认 `projects/<x>/context.md` 出现结构化条目、原文进 `ingest/`。
2. **执行**：给一个小需求 → 确认它先读了 context、产出合理、新坑自动回写且未打断。
2b. **调和**：再摄取一段和已有 context **矛盾**的信息 → 确认系统**停下来问你**该如何处理（而非静默覆盖），旧条目被标 superseded。
3. **润色**：粘一段英文 → 得到润色版；手动改两处喂回 → 确认 `english-profile.md` 新增了对应规则、配对进了 `english-samples/`。
4. **复盘**：跑 `/复盘` → 确认生成当天 review 且认知结构化打标签。聚类/晋升专项：
   - 造 3 天**措辞不同但语义相同**的认知 → 确认在 `recurring.md` 聚成**一条**候选（count=3），而非 3 条。
   - 若这 3 次**同一项目** → 确认**不晋升**（跨情境硬门拦住）；改成**跨 2 个项目** → 确认触发晋升、**停下来问你确认**、写入 `assets/` 标 `已验证可迁移` 并链回证据。
   - 标记一条**单次顿悟为"重要"** → 确认走门二晋升、标 `待验证`。
5. **常驻指针**：新会话里让 skill 先读 `AGENTS.md` + `assets/INDEX.md` + `style/*-profile.md` → 确认核心风格规则/资产摘要被加载、产出"记得你"。
6. **可移植性**：在 **Codex / Cursor / Copilot** 里各跑一次同一关键词（如"摄取"）→ 确认都能读 `AGENTS.md`→对应 skill 并产出一致结果，证明内核不依赖任何单一 agent。
