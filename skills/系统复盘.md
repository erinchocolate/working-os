# skill: 系统复盘 —— 定期把「复盘」对准 working-os 本身（元循环）

**触发**：
- 「系统复盘」→ 对本系统做一次复盘。
- 「系统复盘 <日期>」→ 复盘指定周期。
- 双周或月度提醒出现后，主人说「系统复盘」。

**目的**：循环 A/B 是用系统干活；本 skill 是用系统改系统。定期把摩擦点变成对 `AGENTS.md`、`skills/` 和目录约定的真实改动。

## 先读
1. `AGENTS.md` —— 系统现状与约定。
2. `skills/` 目录列表 —— 有哪些工作流可被改。
3. 最近周期的 `reviews/daily/`、`reviews/weekly/`、`worklog/`、`reviews/recurring.md`。
4. `inbox/`、`inbox/_needs-review/`、`worklog/` 的堆积情况。
5. `assets/INDEX.md`、各 `projects/<项目>/context.md` 的体量。

## 流程一：量飞轮（先诊断，别凭感觉）
统计这一周期的客观信号：
- `inbox/` 是否堆积？`_needs-review/` 有多少待确认？
- `worklog/` 新增几篇？执行后是否真有记录？
- `reviews/daily/` 新增几篇、共几条认知？周复盘有没有跑？
- `assets/` 新晋升几条？候选池 `recurring.md` 涨了多少？
- 各 `projects/` 的 context 有没有变厚（摄取/执行回写有没有真发生）？
- 关键词（摄取/执行/收工/复盘/系统复盘/润色/分享/学习）哪些用过、哪些一次没碰？

一句话判断当前阶段：
- **播种期**：`assets/` 仍接近空、context 薄 → 下周期继续偏向「摄取 + 复盘」。
- **杠杆期**：`assets/` 已有 ≥3 条晋升资产且加载真能抄近路 → 可以多用执行/分享，本 skill 重心转向调优。

## 流程二：三问（核心）
1. 这周期哪一次，加载 context/assets 真的省了时间？
2. 这周期哪一次，用 skill 反而比直接干还累？
3. 这周期有没有一条认知、会议信号或 agent session，系统没接住？

## 流程三：改系统
针对摩擦点：
1. 定位是哪份 `skills/*.md`、`AGENTS.md`、目录约定或提醒脚本造成的别扭。
2. 提出具体改动（删一步、改触发、调先读顺序、加一条原则、改提醒时间等）。
3. 停下来问主人：接受 / 改写 / 否决。确认后才改文件。
4. 改完在系统复盘记录里写清「改了什么 + 为什么」。

## 流程四：记账
写 `reviews/system/<日期>.md`：
```
- 阶段: 播种期 / 杠杆期
- 飞轮信号: inbox N 个 / worklog N 篇 / daily N 篇 / 新晋升 M 条 / context 增量
- 省时刻: 哪次加载帮了忙
- 摩擦点: 哪次别扭 → 改了哪份 skill / 为什么
- 下周期偏向: 一句话
```

## 边界
- 本 skill 可以改 `AGENTS.md`、`skills/`、`scripts/` 和目录约定。
- 不直接改项目事实和个人长期资产；那些仍由「摄取」「执行」「复盘」负责。
