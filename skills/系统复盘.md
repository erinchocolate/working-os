# skill: 系统复盘 —— 每周把「复盘」对准系统本身，让系统越用越顺（元循环）

**触发**：
- 「系统复盘」→ 对本系统做一次每周复盘。
- 「系统复盘 <日期>」→ 复盘指定那一周。

**目的**：循环 A/B 是用系统干活；本 skill 是**用系统改系统**。每周固定问几个问题，把摩擦点变成对 `skills/*.md` 的真实改动——一个用了一个月的 skill 应该和初版长得很不一样。同时判断飞轮转到了哪个阶段，调整下周的偏向。

## 先读
1. `AGENTS.md` —— 系统现状与约定。
2. `skills/` 目录列表 —— 有哪些工作流可被改。
3. `reviews/daily/` 最近一周 + `reviews/recurring.md` —— 这周循环 B 跑得怎么样。
4. `assets/INDEX.md`、各 `projects/<项目>/context.md` 的体量 —— 库厚了没有。

## 流程一：量飞轮（先诊断，别凭感觉）
统计这一周的客观信号：
- `reviews/daily/` 这周新增几篇、共几条认知？
- `assets/` 这周新晋升几条？候选池 `recurring.md` 涨了多少？
- 各 `projects/` 的 context 有没有变厚（执行回写有没有真发生）？
- 六个关键词（摄取/执行/润色/分享/复盘/学习）这周哪些用过、哪些一次没碰？

一句话判断当前阶段：
- **播种期**：`assets/` 仍接近空、context 薄 → 下周继续偏向「摄取 + 复盘」，刻意压低执行/分享期待。
- **杠杆期**：`assets/` 已有 ≥3 条晋升资产且加载真能抄近路 → 下周可放心多用执行/分享，本 skill 重心转向调优。

## 流程二：三问（核心）
1. **这周哪一次，加载 context/assets 真的省了时间？** → 那条路径对了，想怎么强化（写进哪个 skill 的「先读」、补进 INDEX）。
2. **这周哪一次，用 skill 反而比直接干还累？** → 这是摩擦点，进流程三。
3. **这周有没有一条认知，我嘴上说过但系统没接住？** → 当场补进当天 `reviews/daily/` 或候选池。

## 流程三：改 skill（系统进化就发生在这里）
针对流程二第 2 点的每个摩擦点：
1. 定位是哪份 `skills/*.md`（或 `AGENTS.md` 约定）造成的别扭。
2. 提出**具体改动**（删一步、改触发、调先读顺序、加一条原则……）。
3. **停下来问主人**：接受 / 改写 / 否决。确认后才改文件。
4. 改完在记账里写清「改了什么 + 为什么」，留下系统的演化轨迹。
> 边界：本 skill 可以改 `skills/` 和 `AGENTS.md`；不碰 `projects/` 和 `assets/` 的内容（那是循环 A/B 的地盘）。

## 流程四：记账
写 `reviews/system/<日期>.md`：
```
- 阶段: 播种期 / 杠杆期
- 飞轮信号: 这周 daily N 篇 / 新晋升 M 条 / context 增量 …
- 省时刻: 哪次加载帮了忙
- 摩擦点: 哪次别扭 → 改了哪份 skill / 为什么
- 下周偏向: 一句话
```

## 流程五：阶段切换
若本周首次满足「`assets/` ≥3 条晋升资产 且 加载它能改变产出」→ 宣布从播种期进入杠杆期，并在记账里写明触发证据。播种期的低杠杆感是正常的，没到门槛前不要判系统死刑。
