# skill: 执行 —— 基于积累的 context 把需求转成代码/方案

**触发**：主人说「执行 <项目名> <需求>」。

**目的**：复用项目已积累的 context，少走弯路；干活产生的新知识顺手回写，让项目越用越懂自己。

## 先读（重要：先读再动手）
1. `projects/<项目>/context.md` + `decisions.md` + `pitfalls.md` —— 当背景。
2. 常驻指针 `assets/INDEX.md`（主人的通用方法/坑）。
3. 需要时**直接读真实工作仓库的代码与表结构（DDL/schema）**——库里没有真实数据值，可放心读，照真实结构干活最准。`mc-working-os` 本身只承载 context/认知层，**不复制代码**。

## 流程
1. 读完背景，确认理解需求；有歧义先澄清。
2. 检查 `pitfalls.md`：这个需求会不会撞上已知的坑。
3. 产出代码/方案。
4. **副产品自动回写**（干活的 side effect，**不打断主人**）：
   - 新决策/取舍 → **追加**进 `decisions.md`（只追加；旧决策若被取代，标 `已被 X 取代 (日期)`）。
   - 新踩的坑 → 写入 `pitfalls.md`。
   - context 有变化 → 按「摄取」里的调和规则更新 `context.md`。
5. 回写遵循人机边界：新增/更新自动且静默；**矛盾、归档仍停下来问**（但尽量攒到一次工作结束再问，别打断心流）。

## 原则
- 只回写「代码说不出来的东西」（为什么这么做、踩了什么坑、定了什么），不复述代码。
- 执行回写默认全自动不打断；这是 context 三个更新来源里最重要的一条。
