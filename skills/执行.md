# skill: 执行 —— 基于 context + 真实代码仓库把需求转成代码/方案

**触发**：主人说「执行 <项目名> <需求>」「改代码」「实现这个任务」等。

**目的**：复用项目已积累的 context，少走弯路；干活产生的新知识顺手回写，让项目越用越懂自己。

## 先读（重要：先读再动手）
1. `projects/<项目>/project.md` —— 确认项目对应哪些真实代码仓库、复盘时看哪些 git roots。
2. `projects/<项目>/context.md` + `design/decisions.md` + `pitfalls.md` —— 当背景。
3. 常驻指针 `assets/INDEX.md`（主人的通用方法/坑）。
4. 按需求直接读 `project.md` 里的真实代码仓库与表结构（DDL/schema）。`working-os` 只承载 context/认知层，**不复制代码**。

## 流程
1. 读完背景和代码映射，确认理解需求；有歧义先澄清。
2. 检查 `pitfalls.md`：这个需求会不会撞上已知的坑。
3. 到 `project.md` 的 `Code roots` 中读取真实代码，产出代码/方案。
4. **副产品自动回写**（干活的 side effect，尽量不打断主人）：
   - 新决策/取舍 → 追加进 `design/decisions.md`（只追加；旧决策若被取代，标 `已被 X 取代 (日期)`）。
   - 新踩的坑 → 写入 `pitfalls.md`。
   - context 有变化 → 按「摄取」里的调和规则更新 `context.md`。
5. **写执行 worklog**：任务结束时写 `worklog/<日期>-<项目>-<简述>.md`，记录本次任务摘要。

## worklog 格式
```
# <日期> · <项目> · <任务简述>

## Task
- 主人要完成什么。

## Context Used
- 读了哪些 working-os context。
- 读了哪些代码路径。

## Changes / Output
- 改了什么或产出了什么方案。

## Decisions / Pitfalls / Follow-ups
- 新决策、坑、待确认问题。

## Review Inputs
- 复盘时应关注的 git roots、文件或对话线索。
```

## 原则
- 只回写「代码说不出来的东西」（为什么这么做、踩了什么坑、定了什么），不复述代码。
- 新增/更新自动且静默；矛盾、归档、长期资产晋升仍停下来问。
- 执行结束的 worklog 是日复盘的主要输入之一。
