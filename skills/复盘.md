# skill: 复盘 —— 日/周沉淀认知，跨情境复现晋升为认知资产（循环 B）

**触发**：
- 「复盘」→ 复盘当前日期。
- 「复盘 <日期>」→ 复盘指定日。
- 「周复盘」/「weekly review」→ 复盘本周。
- 「复盘 照镜子」→ 跑照镜子流程。

**目的**：从工作、摄取、代码改动和 AI session 里提取认知；反复出现（且跨情境）的认知，提炼成可带走的「认知资产」。

## 先读
1. `assets/INDEX.md`（已有资产）。
2. `reviews/recurring.md`（候选池现状）。
3. 当天或本周的 `worklog/`、`projects/*/ingest/`、`reminders/`。
4. `projects/*/project.md`，用于找到相关代码仓库并读取 git changes。

## 流程一：写当天复盘
1. 确认日期，日期必须写绝对值（如 `2026-07-01`）。
2. 扫当天材料：
   - `worklog/<日期>-*.md`：AI 对话总结、执行任务总结。
   - `projects/*/ingest/<日期>-*.md`：当天摄取的原文。
   - `projects/*/project.md` 中的 `Code roots`：用 `git status`、`git diff`、`git log --since=<日期>` 找当天代码线索。
   - 当天提醒 `reminders/daily/<日期>.md`（若存在）。
3. 若关键材料不足，先问主人一句「今天还有哪件事值得补充？」；主人跳过也可以继续。
4. 写 `reviews/daily/<日期>.md`。每条认知结构化打标签：
   ```
   - type: 认知 / 坑 / 方法 / 偏好
     statement: 一句话认知
     evidence: 今天哪件事触发的
     context: 所属项目/情境
     tags: [关键词]
   ```

## 流程二：候选池增量聚类（不全量重扫历史）
把当天每条认知拿去和 `reviews/recurring.md` 里现有聚类做语义匹配：
- **命中**已有聚类 → `count+1`、加 instance、更新 `last_seen`、若来自新项目则 `contexts` +1、顺手打磨 `canonical`。
- **未命中** → 开一条新候选（`count=1`）。

候选条目格式：
```
- canonical: 规范化的一句话认知（打磨到「可复用原则」高度）
  type: 认知 / 坑 / 方法 / 偏好
  count: N   first_seen: 日期   last_seen: 日期   contexts: [项目A, 项目C]
  instances: [每次原话 + 链回 reviews/daily/<日期>.md]
  status: 候选 / 已晋升 / 已搁置
```

## 流程三：周复盘
1. 读取本周 `reviews/daily/`、`worklog/`、`projects/*/ingest/` 增量和 `reviews/recurring.md`。
2. 写 `reviews/weekly/<周五日期>.md`，总结本周主题、重复出现的认知、项目 context 增量、下周关注点。
3. 找出可能晋升 `assets/` 的候选，但写入资产前必须问主人确认。

## 流程四：晋升判定（平衡型 + 跨情境硬性要求，两扇门）
- **门一·复现型**：`≥3 个不同日子` 且 `跨 ≥2 个情境/项目` 且过非平凡门槛 → 标签 `已验证可迁移`。
- **门二·单次高信号型**：被主人（或你）显式标记「这条重要」的深刻顿悟 → 绕过复现/跨情境，过非平凡门槛即可 → 标签 `待验证`。
- **非平凡门槛**：必须是关于「主人怎么思考/工作」的，不是一时状态或情绪。

## 流程五：晋升（综合提炼 + 停下来问，不是复制原话）
够格的候选：
1. 把聚类综合提炼成一句脱离单次实例的资产。
2. 归类：心智模型 → `assets/mental-models.md`；通用坑 → `assets/pitfalls.md`；长处/盲点 → `assets/mirror.md`。
3. 停下来问主人：接受 / 改写 / 否决 / 搁置。
4. 确认后写进 `assets/`，链回证据，候选标 `已晋升`。
5. 更新 `assets/INDEX.md` 精简索引。

## 流程六：照镜子
读 `assets/` + `reviews/` 全集 → 指出主人还没命名的长处/盲点 → 写入 `assets/mirror.md`，并和主人确认。

## 原则
- 复盘不是流水账，只沉淀可复用认知、坑、方法、偏好。
- 工作事实优先沉淀到项目 context；关于主人如何工作的东西才进 `reviews/` / `assets/`。
- 自动收集材料，长期画像确认后再写。
