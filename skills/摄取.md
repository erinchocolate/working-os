# skill: 摄取 —— 把 inbox/笔记/会议/交流转成结构化项目 context

**触发**：
- 「摄取 inbox」/「处理 inbox」→ 扫描 `inbox/` 里的未处理 Markdown。
- 「摄取 <项目名>」+ 粘贴内容 → 直接摄取到指定项目。
- 「摄取 <项目名> <文件路径>」→ 摄取指定文件。

**目的**：主人只负责捕获，不负责整理；AI 把零散材料沉淀成「代码说不出来的那层」项目知识。

## 先读
1. 常驻指针：`assets/INDEX.md`、`style/*-profile.md`（了解主人）。
2. `projects/*/context.md` 与 `projects/*/project.md`（判断材料属于哪个项目）。
3. 若已确定目标项目，再读该项目的 `context.md`、`design/index.md`、`pitfalls.md`、`open-questions.md`，在已有认识上调和。

## inbox 语义

- `inbox/` 是无分类入口：会议记录、聊天记录、随手笔记、Copilot notes、文档摘录都可以直接扔进来。
- 主人不需要提前写 metadata；AI 在摄取时判断类型、日期、项目、敏感度和路由目标。
- 摄取完成后，原 inbox 文件移动到 `inbox/_processed/<日期>/`，防止重复摄取。
- 只有后台无人值守或一次问题太多时，才把待确认问题写入 `inbox/_needs-review/`；交互摄取时优先当场问主人。

## 流程一：摄取 inbox
1. 扫描 `inbox/` 根目录下的 `.md` 文件，跳过 `_processed/`、`_needs-review/` 和隐藏文件。
2. 对每个文件先做分类判断：材料类型（会议/聊天/笔记/文档摘录/PR 讨论/其他）、可能项目、日期、简述、敏感度。
3. 若项目、敏感度、是否应保存原文等关键 metadata 不确定：
   - 当前是交互会话 → 直接问主人，拿到回答后继续。
   - 当前是后台/无人值守 → 生成 `inbox/_needs-review/<日期>-<简述>.md`，记录问题和原文件路径，然后跳过该文件。
4. 确定项目后，把原文归档到 `projects/<项目>/ingest/<日期>-<简述>.md`。归档文件顶部写 metadata：`source_file`、`captured_at`、`material_type`、`project`、`sensitivity`。
5. 从原文抽取「代码说不出来的东西」，按路由表写入项目 context。
6. 把 inbox 原文件移动到 `inbox/_processed/<日期>/`。
7. 回报：处理了哪些文件、归档到哪里、更新了哪些 context、哪些问题仍待确认。

## 流程二：直接摄取内容/文件
1. **定项目**：项目名不明确就问主人归到哪个项目；新项目则在 `projects/<项目>/` 下建 `project.md`、`context.md`、`pitfalls.md`、`open-questions.md`、`design/`、`ingest/`。
2. **存原文**：原始内容归档进 `projects/<项目>/ingest/<日期>-<简述>.md`（不可变事件流，永不改）。
3. **抽取并路由**：对每条信息判断类型，写入对应文件。

| 信息类型 | 路由目标 | 说明 |
|---|---|---|
| 当前状态/进度 | `context.md`「当前状态 live」 | 高频调和 |
| 怎么操作/workflow | `<主题>-workflow.md` 或既有 workflow 文档 | 操作层面变更 |
| 为什么这么设计 | `design/<子系统>.md` | 按关键词路由到子系统 |
| 明确决策 | `design/decisions.md` | 只追加，旧的标 superseded |
| 踩坑 | `pitfalls.md` | 项目专属坑 |
| 未解决问题 | `open-questions.md` | 插入对应 heading 分组 |
| 新术语 | `glossary.md` | 只增 |

## 调和规则
- **新增**：全新信息 → 加入。
- **更新**：已有条目变了 → 改写并标变更日期。
- **矛盾**：与旧的冲突 → 交互摄取时当场问主人；确认后把旧的标 `superseded (日期)`、写入新的。
- **重复**：已有 → 跳过。
- **归档**：要把某条移出 live 区时 → 交互摄取时当场问主人。

## context.md 分区（简化版）

context.md 只维护三个区：
- 一句话简介（稳定）
- 当前状态 live（高频调和）
- 入口索引（链接到其他文件，偶尔更新）

其他内容已分散到专属文件（design/、glossary、open-questions、pitfalls），不在 context.md 里重复。

## 人机边界
- 自动：分类、归档、抽取、路由、新增、更新、跳过重复。
- 当场问：项目归属不确定、metadata 不确定、敏感度不确定、矛盾、归档、是否晋升长期资产。
- 后台 fallback：无法当场问时，把问题写入 `inbox/_needs-review/`。
