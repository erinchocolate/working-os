# skill: 摄取 —— 把笔记/会议/交流转成结构化项目 context

**触发**：主人说「摄取 [项目名]」并粘贴一段内容（会议记录、和同事的交流、随手笔记）。

**目的**：把零散信息沉淀成「代码说不出来的那层」项目知识，让系统越来越懂这个项目。

## 先读
1. 常驻指针：`assets/INDEX.md`、`style/*-profile.md`（了解主人）。
2. 目标项目的 `projects/<项目>/context.md`、`decisions.md`、`pitfalls.md`（要在已有认识上调和，不是从零写）。

## 流程
1. **定项目**：项目名不明确就问主人归到哪个项目；新项目则在 `projects/<项目>/` 下建 `context.md`、`decisions.md`、`pitfalls.md`、`ingest/`。
2. **脱敏**：不要把真实数据值 / 凭证 / PII 写入文件。代码、表名、字段名、DDL 可正常保留。
3. **存原文**：原始内容**只追加**进 `projects/<项目>/ingest/<日期>-<简述>.md`（不可变事件流，永不改）。
4. **抽取结构化信息**：决策、待办、术语、涉及的人/谁负责什么、开放问题、设计的「为什么」。
5. **调和进 `context.md`**（不是机械追加，对每条做判断）：
   - **新增**：全新信息 → 加入。
   - **更新**：已有条目变了 → 改写 live 条目并标变更日期。
   - **矛盾**：与旧的冲突 → **停下来问主人**怎么处理；确认后把旧的标 `superseded (日期)`、写入新的。
   - **重复**：已有 → 跳过。
   - **归档**：要把某条移出 live 区时 → **停下来问主人**。
6. **回报**：简述哪些进了 context、哪些是矛盾/归档待确认。

## context.md 分区（按更新频率）
一句话(稳定) / 术语表(只增) / 关键人(偶改) / 设计的「为什么」(演进带日期) / 当前状态 live(高频调和) / 开放问题(状态 open→resolved，resolved 归档)。决策入 `decisions.md`、坑入 `pitfalls.md`。

## 人机边界
矛盾、归档 → 停下来问；新增、更新、跳过重复 → 自动。
