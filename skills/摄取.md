# skill: 摄取 —— 把笔记/会议/交流转成结构化项目 context

**触发**：主人说「摄取 [项目名]」并粘贴一段内容（会议记录、和同事的交流、随手笔记）。

**目的**：把零散信息沉淀成「代码说不出来的那层」项目知识，让系统越用越懂项目。

## 先读
1. 常驻指针：`assets/INDEX.md`、`style/*-profile.md`（了解主人）。
2. 目标项目的 `projects/<项目>/context.md`（项目地图）。
3. 按需读 `design/index.md`、`pitfalls.md`、`open-questions.md`（在已有认识上调和，不是从零写）。

## 流程
1. **定项目**：项目名不明确就问主人归到哪个项目；新项目则在 `projects/<项目>/` 下建 `context.md`、`pitfalls.md`、`open-questions.md`、`design/`、`ingest/`。
2. **存原文**：原始内容**只追加**进 `projects/<项目>/ingest/<日期>-<简述>.md`（不可变事件流，永不改）。
3. **抽取并路由**——对每条信息判断类型，写入对应文件：

| 信息类型 | 路由目标 | 说明 |
|---|---|---|
| 当前状态/进度 | `context.md`「当前状态 live」 | 高频调和 |
| 怎么操作/workflow | `deployment-workflow.md` | 操作层面变更 |
| 为什么这么设计 | `design/<子系统>.md` | 按关键词路由到子系统 |
| 明确决策 | `design/decisions.md` | 只追加，旧的标 superseded |
| 踩坑 | `pitfalls.md` | 项目专属坑 |
| 未解决问题 | `open-questions.md` | 插入对应 heading 分组 |
| 新术语 | `glossary.md` | 只增 |

5. **子系统路由关键词**：
   - `design/deployment.md`：bundle, deploy, endpoint lifecycle, app.yaml, target, workspace, trace isolation, app runtime
   - `design/rag-pipeline.md`：retrieval, condensation, citation, location filter, vector search, evaluation, model, RAG
   - 新子系统：当信息不属于已有 design 文件时，新建 `design/<名称>.md` 并在 `design/index.md` 注册

6. **调和规则**（对每条做判断，不是机械追加）：
   - **新增**：全新信息 → 加入。
   - **更新**：已有条目变了 → 改写并标变更日期。
   - **矛盾**：与旧的冲突 → **停下来问主人**怎么处理；确认后把旧的标 `superseded (日期)`、写入新的。
   - **重复**：已有 → 跳过。
   - **归档**：要把某条移出 live 区时 → **停下来问主人**。

7. **回报**：简述哪些进了哪个文件、哪些是矛盾/归档待确认。

## context.md 分区（简化版）

context.md 只维护三个区：
- 一句话简介（稳定）
- 当前状态 live（高频调和）
- 入口索引（链接到其他文件，偶尔更新）

其他内容已分散到专属文件（design/、glossary、open-questions、pitfalls），不在 context.md 里重复。

## 人机边界
矛盾、归档 → 停下来问；新增、更新、跳过重复、路由 → 自动。
